#1 - Admin Panel Setup
What & Why
Build the foundational admin panel for BlockTrade on a new admin-panel branch. The database already has all the necessary infrastructure (is_admin column, fn_admin_update_withdrawal RPC, admin RLS policies on transactions, kyc_submissions table). What is missing is the frontend — a route guard, an admin shell layout, and working pages for the four core admin functions.

Done looks like
Visiting /admin with a non-admin account redirects to / with a "Not authorised" toast
Visiting /admin with an is_admin = true account shows an admin shell with a sidebar distinct from the regular user sidebar
/admin (dashboard) shows four summary stat cards: pending withdrawals count, pending KYC count, total users, and total platform portfolio value
/admin/withdrawals lists all pending withdrawal transactions in a table; each row has Approve and Reject buttons that call the existing fn_admin_update_withdrawal RPC
/admin/kyc lists all pending KYC submissions in a table; each row has Approve and Reject buttons that call a new fn_admin_review_kyc RPC (SQL provided in sql/kyc-admin-review.sql)
/admin/users lists all registered users in a table with their KYC tier, status, and a toggle to grant/revoke admin
All admin pages are fully hidden from regular users via the route guard
A new SQL file sql/kyc-admin-review.sql is provided with the fn_admin_review_kyc function and the required admin RLS policies for kyc_submissions
Out of scope
Email notifications on approval/rejection (Phase 6 of the roadmap)
Activity log integration
Any changes to the regular user-facing app
Tasks
Create the admin-panel branch — Checkout a new branch named admin-panel from the current main state before making any file changes.

AdminRoute guard — Create src/components/AdminRoute.jsx that wraps ProtectedRoute, checks user.is_admin from the Supabase users table, and redirects non-admins to / with a sonner toast error.

Admin API layer — Create src/lib/api/admin.js with functions for: fetching pending withdrawals, calling fn_admin_update_withdrawal, fetching pending KYC submissions, listing all users, and updating user admin/status flags directly.

Admin layout and navigation — Create src/pages/admin/AdminLayout.jsx with its own sidebar (dark theme, clearly labelled "Admin Panel") containing links to Dashboard, Withdrawals, KYC Review, and Users. Wire AdminRoute and AdminLayout into src/App.jsx under /admin.

Admin Dashboard page — Create src/pages/admin/AdminDashboard.jsx showing four stat cards (pending withdrawals, pending KYC, total users, total portfolio value) pulled live from Supabase.

Withdrawal review page — Create src/pages/admin/AdminWithdrawals.jsx with a filterable table of all withdrawal transactions. Each row has an Approve button and a Reject button that opens a confirmation modal with a required reason field.

KYC review page — Create src/pages/admin/AdminKyc.jsx with a table of all KYC submissions. Each row opens a detail modal showing document image URLs and has Approve/Reject buttons that call fn_admin_review_kyc.

User management page — Create src/pages/admin/AdminUsers.jsx with a table of all users. Columns: display name, email, KYC tier, account status, joined date, admin toggle.

KYC admin SQL migration — Create sql/kyc-admin-review.sql with the fn_admin_review_kyc stored procedure (updates kyc_submissions, propagates to users.kyc_verified) and the two admin RLS policies for kyc_submissions.
